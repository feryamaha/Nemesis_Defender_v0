Defense in Depth · Rust · v2.0

Nemesis

Enforcement determinístico contra malware de supply-chain e abuso de agentes LLM em ambientes de desenvolvimento.

versão 2.0 workspace 8.2.0 linguagem Rust plataforma Linux + fallback licença Apache 2.0
⚠ Aviso honesto de escopo

O Nemesis é um sistema de enforcement robusto, em camadas independentes, para elevar significativamente o custo de um ataque. Ele não é — e nenhum sistema de segurança é — "impenetrável". Esta doc descreve o que ele faz, contra qual modelo de ameaça, e o que ele não faz.

Um sistema de enforcement para fluxos de desenvolvimento assistido por LLM, escrito em Rust. Detecta e bloqueia padrões conhecidos de malware de supply-chain e comandos destrutivos antes que executem.

Não é um linter genérico. O foco é o contexto de desenvolvimento guiado por agentes de IA, onde um output aparentemente inócuo pode esconder manipulação de manifests, decode-then-exec de payloads codificados, esteganografia Unicode (CVE-2021-42574), prompt injection, exfiltração de credenciais e malware com time-gating.

O problema é real e atual: em 2025 foram publicados centenas de milhares de pacotes maliciosos no npm, e campanhas como Shai-Hulud comprometeram pacotes com bilhões de downloads semanais. O Nemesis ataca a janela em que esse código tenta executar na máquina do desenvolvedor.

02 — Gênese

Nasceu da dor, não da prancheta

O Nemesis não foi projetado num quadro de segurança. Cresceu de uma dor concreta e repetida dentro de projetos reais de produção — e evoluiu em três fases.

Fase 1 · regras em markdown

Convenções escritas na IDE

No início eram só regras em texto, tentando conter os mesmos erros que se repetiam: hooks condicionais, setState em useEffect, any, tipagens duplicadas, CSS fora do tailwind.config, lógica de negócio em arquivos de UI. A lição: regra escrita é input de contexto — o modelo lê, entende, e ainda assim executa o padrão neural de "resolver rápido". Instrução não é enforcement.

Fase 2 · Node/TypeScript

O linter automático

As convenções viraram hooks que rodam e bloqueiam de fato. O primeiro salto foi um linter automático: em vez de "por favor não use any", o sistema passou a barrar o anti-pattern antes de entrar no repositório. Determinístico, não negociável.

Fase 3 · Rust + segurança

Do linter ao escudo de kernel

Reescrito em Rust, com o foco virando 100% segurança: deny-list de comandos, scanner de conteúdo, detecção de supply-chain e a camada eBPF no kernel. A mesma filosofia que barrava any passou a barrar rm -rf destrutivo e exfiltração de credenciais.

A linha que conecta as três fases: as regras são rígidas porque a dor das violações foi real e recorrente. Só o enforcement que roda fora da inferência (lint-time, PreToolUse, kernel) fecha o gap entre regra e comportamento.

Nota de classificação honesta

A camada de governança de workflow (orquestra como o agente trabalha) opera por engenharia de prompt e atinge, por estimativa do autor, ~80% de eficácia — depende, em parte, da inferência. Já o enforcement (deny-list, AST, eBPF) é determinístico e não depende da cooperação do modelo. A governança guia; o enforcement obriga.

03 — Por que existe

Impedir o comando destrutivo, antes dele rodar

A função mais importante é cotidiana: impedir que um agente execute, sem intenção, um comando que destrói o projeto — rm -rf no lugar errado, git reset --hard que apaga trabalho não commitado, sobrescrita de config.

Quem programa com agentes LLM sabe que isso não é hipótese: o modelo, tentando "ajudar", roda um comando destrutivo porque interpretou mal o contexto. O Nemesis intercepta antes da execução e exige confirmação humana. Vale para o erro involuntário (o caso comum) e para o código malicioso.

Modelos operam por inferência probabilística — pedir "não rode comandos destrutivos" é instrução que o modelo pode ignorar, errar ou rotear ao redor. O enforcement determinístico antes da execução é categoricamente diferente: não importa se o modelo foi enganado ou apenas errou, a camada bloqueia. Ferramentas reativas (linters, CI/CD, SAST) detectam depois; o Nemesis bloqueia antes. Complementar a elas — não substituto.

04 — Linha de autoridade

O modelo opera; ele não destrói

O coração do design é uma divisão clara do que a IA pode e não pode fazer:

  • A IA usa as ferramentas nativas de escrita e edição da IDE — mas só dentro do scaffold do projeto.
  • Pode ler alguns arquivos sensíveis quando o trabalho exige, mas não editar nem apagar nenhum deles.
  • Exclusão, sobrescrita fora do escopo, reset, comando destrutivo: prerrogativa exclusivamente humana, não delegável.

Essa fronteira não depende da boa vontade do modelo — é imposta pelas camadas abaixo. O "zero incidente" em produção não vem do modelo "se comportar", mas da arquitetura nunca lhe dar essa autonomia.

05 — Arquitetura

Três camadas independentes

01

Pretool Hook

Win · macOS · Linux

Intercepta comandos antes de Bash.run(), valida contra deny-list.json e regras de escopo. Bloqueio é hard-gate: sem aprovação humana, exit code 2.

02

Nemesis Defender

Win · macOS · Linux

Scanner de conteúdo com 6 layers — AST (tree-sitter) → byte → regex → denylist → entropia → decoder recursivo (máx. 3 níveis) — e 12 visitors cobrindo os vetores catalogados.

03

eBPF Kernel LSM

Linux apenas

A camada com mais "dentes": bloqueia syscalls perigosas no kernel (mount, ptrace, kexec_load, delete_module → EPERM). É uma camada adicional, não plano B de falha — opera em paralelo, garantindo que mesmo um vetor sofisticado que contorne as de cima ainda esbarre no kernel. Tem deny-list própria, atrelada à arquitetura (no Cargo.toml do módulo). Em produção (violations.log) registra bloqueios de rm, shred, dd, kill, mount, nc e runtimes (python3, perl).

07 — Customização

Tudo é configurável — e só por humano

Princípio central: não há regra hardcoded. Os hooks são agnósticos e canônicos — se um padrão está na deny-list, é bloqueado; se não está, passa. Não existe lógica de decisão escondida no código do hook.

  • Qualquer vetor, regex ou regra pode ser desativado, relaxado ou customizado editando os arquivos de deny-list.
  • Ampliar a cobertura (novo vetor) é adicionar uma entrada — não recompilar lógica.
  • Apenas um humano faz essas mudanças. O modelo não altera a própria gaiola: os arquivos de config estão sob absolute_block.

Duas exceções exigem conhecimento mais profundo: a camada eBPF tem sua lista de syscalls atrelada à arquitetura (no Cargo.toml), e os visitors do Defender são código Rust de análise — ampliá-los exige Rust e segurança. As deny-lists, por outro lado, são simples de ajustar.

08 — Controle de acesso

Quem pode ler/editar o quê

O arquivo denylist-folder-files.json define, sob controle exclusivamente humano, o que o agente pode tocar. É aqui que a linha de autoridade vira prática, em dois níveis:

  • absolute_block — bloqueio total (leitura + escrita + deleção): .env, .ssh/id_rsa, .bashrc/.zshrc, os settings/hooks de cada IDE (.claude/, .cursor/, .windsurf/) e o próprio .nemesis/.
  • write_block — leitura permitida, escrita/edição bloqueada: package.json, next.config.js, eslint.config.mjs, .gitignore e os logs.
  • allowed_exceptions — o scaffold liberado (ex.: /src/), onde o agente escreve e edita à vontade.
Bypass é humano

Só um humano comuta essas permissões. Precisa relaxar uma restrição para manutenção? O bypass é humano e explícito — o agente nunca promove a si mesmo. E, independente de qualquer permissão de leitura/escrita, comando destrutivo permanece sempre proibido para a IA.

09 — Qualidade

Camada AST-linters

A primeira versão do Nemesis nasceu como controle de qualidade. Na evolução de Node/TS para Rust, o foco virou 100% segurança — e a qualidade foi reduzida ao que afeta ou pode afetar segurança e estabilidade: exposição de API/credenciais, aninhamento de tags que quebra build/deploy, falhas graves que derrubam a aplicação.

Existem deny-lists de qualidade por linguagem, cada regra apontando para um arquivo .md:

  • Rust — chain de 3+ unwrap(), unsafe em lib, panic!()/process::exit() em lib, println! em lib.
  • Python — eval(), exec(), pickle.loads(), os.system()/shell=True, f-string com SQL, yaml.load() inseguro, MD5.
  • Java — Runtime.exec(), reflection dinâmica, SQL por concatenação, deserialização.
  • Go — unsafe.Pointer, panic() público, defer sem checar erro, SQL via Sprintf.
  • Genérico — credenciais hardcoded (OWASP A02), secrets, debug output, TODO/FIXME.
Honestidade sobre esta camada

A camada ast-linters está ativa, mas em amadurecimento — precisa de ajustes. E não substitui ESLint ou Biome: é um complemento de bloqueio em runtime para o subconjunto que toca segurança e estabilidade, não um linter completo. Use suas ferramentas de lint normalmente; o Nemesis apenas adiciona a barreira de bloqueio.

10 — Validação

Três tipos de evidência

Da mais forte para a mais fraca. A ordem é deliberada: a evidência que mais importa é a que mais se parece com o uso real.

1 · Track record em produção — mais forte

O Nemesis nasceu em projetos reais de produção (que não podem ser divulgados) e roda há ~1 ano em desenvolvimento ativo, com agentes LLM operando sobre o código. Nesse período, nenhum agente executou um comando destrutivo nem acessou arquivo sensível para edição/exclusão sem permissão. Não é laboratório — é o caso de uso real.

2 · Execução adversarial por agente real

Em sessões registradas (Windsurf no Linux e macOS, Cursor), agentes receberam suítes de ataque disfarçadas de "scaffolding" e tentaram executá-las tarefa a tarefa. O Nemesis bloqueou cada uma no ponto de execução (no Write, no comando aceito), com mensagens específicas. É o teste correto da tese: assume que o modelo pode ser enganado e mede se o enforcement segura. Segurou.

3 · Suíte de vetores sintéticos — apoio

  • O que cobre: os vetores comuns — rm -rf, curl | bash, postinstall malicioso, exfiltração de .env. Onde o dano real acontece.
  • O que NÃO prova: completude ou invulnerabilidade. Suíte do autor; passar nela é o piso esperado.

Os logs: três camadas independentes

  • Camada 3 (eBPF): +2.000 bloqueios reais com "layer":"ebpf", permission_denied — no kernel, independente de deny-list em userspace.
  • Camada 2 (classificador): vereditos nomeados ([MALICIOUS] · reverse_shells) com evidência capturada e instrução de correção.
  • Correlação: detecta brute force ("N tentativas em 300s") e padrões compostos ("leitura de sensível + comando de rede"). Detecção comportamental, acima de regex simples.
Um bypass encontrado e corrigido

A evidência mais útil veio de um adversário real. Após uma manutenção em que se esqueceu de readicionar comandos à deny-list, um agente contornou a Camada 1 por comandos fora da lista. O gap foi identificado e refechado. Lição: surgiu do processo de manutenção, não do design — mitigado com testes de regressão. No Linux, a Camada 3 (eBPF) opera independente da deny-list. Bypasses são o ativo de validação mais valioso, não vergonha a esconder.

11 — Modelo de ameaça

O que protege, o que não protege

◆ Projetado para mitigar
  • Comandos destrutivos por engano do LLM — rm -rf, git reset --hard, sobrescrita de config
  • Supply-chain via manifests (npm, Cargo, PyPI, RubyGems, Composer, lockfiles)
  • Payloads ofuscados — encoding, Unicode, time-gating
  • Exfiltração de credenciais e prompt injection
◆ NÃO protege contra
  • Vetores não contemplados — fora da lista de visitors/patterns
  • Bypass das camadas 1–2 fora do Linux (sem eBPF)
  • Reverse engineering do binário distribuído
  • Atacante root capaz de descarregar daemon/eBPF
  • Ataques fora do escopo command/file/manifest

O Nemesis complementa SAST, linters e CI/CD — não os substitui.

12 — Prior art

Como se compara ao que já existe

Por honestidade: a técnica central tem prior art maduro. Isso não diminui o projeto — situa ele.

  • eBPF/LSM para enforcement é consolidado no cloud-native (KubeArmor, Tetragon/Cilium, Falco). A Camada 3 aplica a mesma classe de técnica.
  • Guardrails para agentes LLM é categoria estabelecida (LlamaFirewall, LLM Guard, NeMo Guardrails, Lakera Guard).
  • Enforcement determinístico em runtime é pesquisa ativa (ex.: AgentSpec, ICSE '26).

O recorte do Nemesis — máquina de dev local, agnóstico de IDE, interceptando comandos do coding agent e escaneando supply-chain no install, para o dev individual — é menos coberto pelas opções acima, que focam em cloud/Kubernetes ou no gateway de LLM. Nicho real, não espaço vazio.

13 — Uso

Build e comandos

Pré-requisitos: Rust 1.70+, Cargo. Linux para a Camada 3 (eBPF); macOS/Windows rodam Camadas 1–2.

# Build do workspace completo
cd .nemesis
cargo build --release --workspace

# Escanear um arquivo
nemesis-defender --scan /caminho/arquivo.rs

# Daemon (filesystem watcher)
nemesis-defender --ensure-daemon
nemesis-defender --stop

# Validar / testar escopo
nemesis-validate /caminho/arquivo.ts
nemesis-scope   /caminho/arquivo.rs

IDEs suportadas via hooks: Claude Code, Codex, Cursor, VS Code, Windsurf, OpenClaude. A biblioteca Rust (nemesis-defender) é agnóstica de IDE.

14 — Workflow

Processo de desenvolvimento ≠ regra do Nemesis

Versões antigas incluíam um enforcement de workflow sequencial (.nemesis/workflow-enforcement/) que impedia o modelo de pular etapas — spec → plano → análise de regras —, reduzindo entropia e dívida técnica no projeto onde o Nemesis nasceu.

Esse modelo era engessado e específico daquele processo. Foi migrado para skills com pipeline mais probabilístico, que dão liberdade à criatividade do modelo.

Importante

Processo de desenvolvimento é de cada equipe — não é uma regra do Nemesis. O que o Nemesis impõe é o enforcement de segurança e qualidade (AST + deny-list + eBPF + pretool), que permanece ativo independentemente do fluxo de trabalho adotado.
