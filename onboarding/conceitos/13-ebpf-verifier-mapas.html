eBPF: o verifier, os mapas e o BPF-LSM

eBPF permite carregar pequenos programas no kernel que são verificados quanto à segurança antes de rodar. O Nemesis usa eBPF para impor políticas de exec e de rede sem escrever um módulo de kernel.

Em 30 segundos

eBPF é como "JavaScript do kernel", só que com um verifier que prova que o programa termina e não acessa memória inválida antes de aceitá-lo. Programas eBPF guardam estado em mapas (hash, array, ring buffer, LPM trie) e podem ser anexados a hooks de segurança (BPF-LSM). O Nemesis usa três tipos de mapa para comandos bloqueados, eventos e allowlist de IP.

Do zero

Escrever um módulo de kernel é arriscado: um bug derruba a máquina. eBPF resolve isso com um verifier: antes de carregar, o kernel analisa o bytecode e rejeita qualquer programa que possa fazer loop infinito, ler memória fora dos limites, ou usar ponteiros não checados. Como o programa não pode manter estado entre execuções na pilha, ele usa mapas: estruturas de dados compartilhadas entre o kernel e o userspace.

Por que importa

O verifier dá uma garantia rara: código no kernel que é seguro por construção. Os mapas permitem que o userspace (o loader do Nemesis) configure dinamicamente o que bloquear sem recompilar nada.

Como o Nemesis aplica

O programa BPF declara os mapas que usa. Veja os três tipos centrais:

📄 .nemesis/ebpf-kernel/ebpf/nemesis-block.bpf.c:20-64 
struct { __uint(type, BPF_MAP_TYPE_HASH); ... } blocked_commands;   // comandos negados
struct { __uint(type, BPF_MAP_TYPE_RINGBUF); ... } events;          // telemetria p/ userspace
struct { __uint(type, BPF_MAP_TYPE_LPM_TRIE); ... } egress_allow_v4; // CIDR allowlist (IPv4)
struct { __uint(type, BPF_MAP_TYPE_LPM_TRIE); ... } egress_allow_v6; // CIDR allowlist (IPv6)
struct { __uint(type, BPF_MAP_TYPE_ARRAY); ... } egress_enforce;    // flag observar/impor

Note os loops com #pragma unroll e limites fixos (BPF_PATH_LEN): é exigência do verifier, que rejeita loops de tamanho não comprovável. O ring buffer publica cada bloqueio para o userspace registrar, sem que o programa BPF precise fazer I/O.

A allowlist usa LPM trie (longest-prefix match), a estrutura natural para casar um IP contra regras CIDR como 10.0.0.0/8. A lógica de parsing das regras é Rust testável:

📄 .nemesis/ebpf-kernel/src/egress.rs — parsing puro do CIDR:porta (testável por cargo test)

Nuances e armadilhas

  • O verifier limita drasticamente o que o programa pode fazer: sem alocação dinâmica, sem loops ilimitados, stack pequena. Daí o BPF_PATH_LEN 128 menor que o caminho real.
  • O self-audit do CI proíbe commitar o objeto compilado .bpf.o: ele deve ser buildado da fonte, para não virar um alvo de troca não-revisável.
  • A decisão real acontece no kernel; o Rust apenas popula os mapas. Separar "lógica pura testável" de "efeito no kernel" é um padrão de design importante aqui.

Auto-checagem

O que o verifier garante?

Que o programa termina (sem loops infinitos) e não acessa memória inválida — segurança por construção antes de carregar no kernel.

Por que mapas em vez de variáveis globais?

Programas eBPF não mantêm estado na pilha entre execuções; os mapas são o estado compartilhado e a ponte com o userspace.

Por que um LPM trie para egress?

Porque a allowlist é em CIDR; LPM faz o casamento por prefixo mais longo, ideal para redes como 192.168.0.0/16.
