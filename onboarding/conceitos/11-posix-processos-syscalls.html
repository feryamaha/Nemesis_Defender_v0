POSIX, processos, syscalls e stdin/stdout/stderr

O Nemesis é um conjunto de processos Unix clássicos: lêem da entrada padrão, decidem, e comunicam o veredito por código de saída e pela saída de erro. Entender esse modelo é entender como o hook conversa com a IDE.

Em 30 segundos

A IDE envia um JSON pela stdin descrevendo a ação pretendida. O hook lê, analisa, escreve a razão na stderr e sai com um exit code. O daemon é um processo de longa duração lançado como filho desacoplado. Tudo são syscalls: pedidos ao kernel para ler, escrever, criar processos.

Do zero

Um processo é um programa em execução, com seu próprio espaço de memória e três fluxos padrão: stdin (0, entrada), stdout (1, saída), stderr (2, erros). Uma syscall é a fronteira entre o programa e o kernel: criar um processo (fork/execve), ler um arquivo (read), enviar um sinal (kill). POSIX é o padrão que define essa interface em sistemas Unix.

Por que importa

Como o hook fala stdin/stderr/exit-code, ele é agnóstico de IDE: qualquer ferramenta que respeite o protocolo de hook funciona. O daemon precisa sobreviver ao processo que o lançou; por isso é desacoplado dos fluxos padrão.

Como o Nemesis aplica

O hook lê o payload da IDE diretamente da stdin (buffer de 128 KiB):

📄 .nemesis/hooks/nemesis-pretool-check-unix.rs:1037-1040 
let mut buffer = vec![0u8; 131072];
if let Ok(n) = io::stdin().read(&mut buffer) {
    stdin_peek = String::from_utf8_lossy(&buffer[..n]).to_string();
}

O daemon é lançado como processo filho com os três fluxos redirecionados para /dev/null — desacoplado, sobrevive ao pretool (fire-and-forget):

📄 .nemesis/nemesis-defender/src/main.rs:106-113 
std::process::Command::new(&exe).arg("--daemon")
    .current_dir(&project_root)
    .stdin(Stdio::null()).stdout(Stdio::null()).stderr(Stdio::null())
    .spawn()

Para parar o daemon, o CLI usa o sinal SIGTERM via kill -TERM <pid>, lendo o PID de um arquivo:

📄 .nemesis/nemesis-defender/src/main.rs:128-138 — --stop

Nuances e armadilhas

  • O hook escreve a razão na stderr (não stdout) porque algumas IDEs interpretam o stdout como dado estruturado (ex.: Antigravity recebe JSON de decisão no stdout).
  • from_utf8_lossy é deliberado: input malformado não derruba o hook, vira caracteres de substituição.
  • Resolução de caminho sobe até o ancestral .nemesis em vez de contar níveis fixos — robusto a layouts diferentes (dev vs distribuído).

Auto-checagem

Por onde a IDE envia a ação pretendida ao hook?

Pela stdin, como JSON. O hook lê com io::stdin().read().

Por que o daemon redireciona stdin/stdout/stderr para null?

Para desacoplar do processo-pai (o pretool), que termina imediatamente. Assim o daemon sobrevive como serviço de longa duração.

Qual syscall encerra o daemon?

O envio do sinal SIGTERM (kill -TERM), com o PID lido do arquivo de PID.
