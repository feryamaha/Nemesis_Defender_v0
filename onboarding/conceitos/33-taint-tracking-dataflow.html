Taint tracking e data-flow (fonte → sink)

Taint tracking segue o caminho de um dado sensível pelo código: da origem (fonte) até onde ele é usado de forma perigosa (sink), mesmo passando por variáveis intermediárias.

Em 30 segundos

Um atacante fragmenta o ataque para escapar de regex: lê um token numa linha, copia para outra variável, envia em uma terceira. O taint tracker marca a variável de origem como "tainted" e propaga essa marca pelas atribuições, disparando quando ela chega a um sink de rede/exec. É análise de fluxo de dados.

Do zero

Fonte (source): onde dado não confiável/sensível entra (env var de credencial, leitura de arquivo de segredo). Sink: onde usá-lo é perigoso (eval, fetch, subprocess). Propagação: o dado se move por atribuições, concatenações, templates. Taint tracking "tinge" a fonte e segue a tinta até o sink. É a base de ferramentas como CodeQL.

Por que importa

Regex vê uma linha por vez. Se o payload está espalhado em três linhas, nenhuma linha isolada parece maliciosa. O fluxo de dados conecta as peças. É o que pega o ataque "leio o segredo aqui, mando ali" que uma assinatura simples perderia.

Como o Nemesis aplica

O tracker roda em três passadas: coleta fontes, propaga o taint por atribuições, e detecta a chegada a um sink:

📄 .nemesis/nemesis-defender/src/visitors/taint_tracker.rs:243-327 — scan_js 
// Pass 1: coletar fontes tainted (env credential, file read sensível)
// Pass 2: propagar taint por atribuição/concat/template
if let Some(new_var) = detect_taint_propagation(line, &tainted_vars) { /* ... */ }
// Pass 3: detectar variável tainted chegando a um sink (exec/rede)
if let Some(v) = detect_taint_in_sink(line, line_num, &tainted_vars,
        NETWORK_SINK_PATTERNS, "network", &source_type) { violations.push(v); }

Quando a fonte é credencial e o sink é rede, o veredito é o mais grave (exfil via variável). O visitor taint_tracker é confirmatório (Tier A): a corroboração está embutida (exige fonte + propagação + sink), então 1 hit já confirma.

Nuances e armadilhas

  • Escopo declarado (v1): intra-arquivo, intra-função; fontes de credencial; propagação por atribuição/concat/template; sinks de exec/rede. O próprio módulo documenta esse "escopo mínimo viável".
  • É determinístico: o fluxo existe ou não existe, sem heurística probabilística. Isso o qualifica como Tier A.
  • Para reduzir falsos positivos, nomes de variáveis sensíveis (token/secret/key…) reforçam o sinal; é rodado como passada full-content após a travessia AST.

Auto-checagem

Fonte e sink, em uma frase cada?

Fonte: onde o dado sensível nasce. Sink: onde usá-lo é perigoso (rede/exec).

Por que taint pega o que regex não pega?

Porque conecta linhas: o payload pode estar fragmentado em várias atribuições; o fluxo de dados liga origem ao destino.

Por que o taint_tracker é Tier A (confirmatório)?

Porque a corroboração é embutida: ele exige fonte + propagação + sink coexistindo, um padrão determinístico de alta confiança.
