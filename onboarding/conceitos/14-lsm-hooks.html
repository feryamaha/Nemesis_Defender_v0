LSM (Linux Security Modules) e os hooks lsm/*

LSM é o framework do kernel que insere pontos de decisão de segurança em operações sensíveis. SELinux e AppArmor o usam; com BPF-LSM, o Nemesis também.

Em 30 segundos

Em pontos-chave (executar binário, abrir socket, ler arquivo), o kernel chama um hook LSM e pergunta "isto é permitido?". Retornar 0 permite; retornar um erro negativo (-EPERM) nega. O Nemesis registra dois hooks: lsm/bprm_check_security (exec) e lsm/socket_connect (rede).

Do zero

Antes do LSM, política de segurança era espalhada e difícil de estender. O LSM padronizou: o kernel expõe dezenas de hooks, e módulos de segurança "plugam" funções neles. Historicamente os módulos eram compilados no kernel (SELinux, AppArmor, Smack). O BPF-LSM permite anexar programas eBPF a esses hooks em tempo de execução, sem recompilar o kernel, com a segurança do verifier.

Por que importa

Os hooks LSM são o lugar canônico para mediar segurança no Linux. Como eles ficam no caminho da operação (a syscall espera a decisão), são um ponto de enforcement real, não apenas observação.

Como o Nemesis aplica

Dois programas, dois hooks. O de rede impõe deny-by-default contra a allowlist, mas só quando o modo enforce está ligado e o processo é do agente:

📄 .nemesis/ebpf-kernel/ebpf/nemesis-block.bpf.c:173-224 
SEC("lsm/socket_connect")
int BPF_PROG(nemesis_check_connect, struct socket *sock, struct sockaddr *address, int addrlen, int ret)
{
    if (ret != 0) return ret;                 // respeita LSM anterior
    u8 *enforce = bpf_map_lookup_elem(&egress_enforce, &zero);
    if (!enforce || *enforce == 0) return 0;  // modo observação
    if (!egress_is_agent()) return 0;         // só o agente
    // ... casa IP/porta na allowlist LPM ...
    if (val && (val->port == 0 || val->port == dport)) return 0; // permitido
    egress_emit("ipv4");
    return -EPERM;                            // negado
}

O kernel precisa ter bpf na lista de LSMs ativos; o pretool verifica isso lendo /sys/kernel/security/lsm antes de subir o daemon eBPF.

📄 .nemesis/hooks/nemesis-pretool-check-unix.rs:948-951

Nuances e armadilhas

  • Famílias não-IP (AF_UNIX, etc.) são permitidas: não são egress de rede. O hook só decide sobre AF_INET/AF_INET6.
  • O hook respeita a decisão de LSMs anteriores na cadeia (if (ret != 0)): o Nemesis coopera com SELinux/AppArmor, não os atropela.
  • Landlock também é um LSM, mas de outra natureza (sandbox aplicada pelo próprio processo). Ver capabilities, cgroups, Landlock.

Auto-checagem

Quais dois hooks LSM o Nemesis usa?

lsm/bprm_check_security (execução de binário) e lsm/socket_connect (conexão de rede).

O que diferencia "observação" de "enforce" no egress?

A flag no mapa egress_enforce: 0 = só registra; 1 = impõe deny-by-default. Em observação, o hook retorna 0 (permite) sempre.

Como o pretool sabe se o kernel suporta BPF-LSM?

Lendo /sys/kernel/security/lsm e checando se contém bpf.
