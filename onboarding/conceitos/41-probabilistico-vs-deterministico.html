Instrução probabilística vs enforcement determinístico

A tese central do Nemesis: você não contém um sistema probabilístico com mais instruções probabilísticas. Contém com um controle determinístico, fora do modelo.

Em 30 segundos

Um LLM é probabilístico: dada a mesma situação, pode agir diferente, e pode ser persuadido a ignorar regras (prompt injection). Pedir "não rode comandos perigosos" no system prompt é uma instrução que às vezes é seguida. Um hook que sai com exit 2 é determinístico: sempre nega, independente do que o modelo "quis".

Do zero

Determinístico: mesma entrada → mesma saída, sempre. Código comum é assim. Probabilístico: a saída é uma amostra de uma distribuição; um LLM gera tokens com aleatoriedade e é sensível a fraseado, contexto e ataques. Misturar os dois numa defesa significa que sua garantia é tão forte quanto o elo mais fraco, e o elo probabilístico não dá garantia.

Por que importa

Toda a indústria tentou "alinhar" modelos com instruções. Mas instrução é texto, e texto pode ser sobrescrito por outro texto (injeção). A única forma de uma garantia dura é tirar a decisão do modelo: um processo separado, em código auditável, que decide antes da ação. Esse é o salto conceitual do projeto.

Como o Nemesis aplica

O veredito de conteúdo é puramente determinístico: regras fixas, regex linear, limiares constantes, sem nenhuma chamada a modelo. A severidade é uma função pura das violações:

📄 .nemesis/nemesis-defender/src/lib.rs:420-443 — compute_severity 
fn compute_severity(violations: &[DefenderViolation]) -> Severity {
    if violations.iter().any(|v| CONFIRMATORY_VISITORS.contains(&v.visitor.as_str())) {
        return Severity::Malicious;            // 1 sinal confirmatório basta
    }
    // conta TIPOS distintos de sinal corroborante
    match distinct.len() { 0 => Clean, 1 => Suspicious, _ => Malicious }
}

Não há aleatoriedade, nem rede, nem LLM nessa decisão. Mesmo arquivo, mesmo veredito, sempre, e por isso o pentest é reprodutível e o gate de CI confiável. Ver self-audit.

Nuances e armadilhas

  • Determinismo não é onisciência: o scanner pode ter falsos negativos. O ponto é que sua decisão é confiável e auditável, não que ele veja tudo.
  • O Nemesis não tenta consertar o modelo; ele assume que o modelo pode ser comprometido e o cerca com controles externos. Isso é threat modeling honesto.
  • O modelo de IA ainda é útil para escrever código; só não é confiável para se auto-policiar. A defesa fica fora dele.

Auto-checagem

Por que não basta um bom system prompt?

Porque é instrução probabilística: pode ser ignorada por amostragem ou sobrescrita por prompt injection. Não dá garantia dura.

O que torna o veredito do Nemesis determinístico?

Regras fixas, regex linear, limiares constantes, sem aleatoriedade nem chamada a modelo: compute_severity é função pura.

O Nemesis tenta "alinhar" o modelo?

Não. Ele assume que o modelo pode falhar/ser comprometido e impõe controle externo determinístico.
