DevSecOps: SLSA, gate de self-audit, CODEOWNERS e regras embutidas

Segurança não para no código: o processo de build e release é parte da superfície. O Nemesis impõe atestação de proveniência, um gate de integridade em todo PR, e regras que vivem dentro do binário.

Em 30 segundos

O CI tem um workflow self-audit que falha o PR se: alguma GitHub Action não estiver fixada por SHA, o Cargo.lock não estiver commitado, houver .bpf.o commitado, alguma CVE de dependência, ou o pentest não passar. As denylists de bloqueio são embutidas no binário (tamper-proof). E o release gera atestação SLSA de proveniência.

Do zero

DevSecOps integra segurança ao ciclo de desenvolvimento (não como etapa final). SLSA é um framework para garantir a integridade da cadeia de build (proveniência: quem buildou o quê, a partir de qual fonte). CODEOWNERS define quem precisa revisar mudanças em arquivos críticos. Pin por SHA evita que uma Action de terceiro seja trocada sob você.

Por que importa

Uma ferramenta de segurança comprometida na build é pior que nenhuma: ela tem a confiança do usuário. O gate de CI torna um comprometimento da fonte/cadeia caro e ruidoso, não silencioso, e as regras embutidas impedem que alguém "relaxe" a denylist editando um arquivo solto.

Como o Nemesis aplica

O gate roda em todo PR e push para main e termina exigindo o pentest aprovado:

📄 .github/workflows/self-audit.yml:36-89 
- name: Actions devem ser fixadas por SHA (anti supply-chain de CI)
- name: Cargo.lock deve estar commitado
- name: Nenhum binario de kernel (.bpf.o) commitado
- name: Auditoria de CVEs (cargo audit)
- name: GATE — Pentest do Nemesis deve estar APROVADO (FAIL=0)
  run: bash .nemesis/pentest-nemesis-control/nemesis-defender/run-pentest.sh

As denylists de bloqueio são compiladas dentro do binário do hook via include_str!, então não há arquivo editável para um atacante afrouxar:

📄 .nemesis/hooks/nemesis-pretool-check-unix.rs:711-721 
const EMBEDDED_DENY_LIST: &str = include_str!("../denylist/deny-list.json");
const EMBEDDED_FOLDER_FILES: &str = include_str!("../denylist/denylist-folder-files.json");

A única superfície editável por humano é a allowlist do cliente; o release publica binários com atestação SLSA (.github/workflows/release.yml).

Nuances e armadilhas

  • O self-audit não se autovalida (trusting-trust): só vale acoplado a CODEOWNERS + branch protection. Ver limitações.
  • "Republicar é o que chega ao usuário": correções só alcançam instalações distribuídas após rebuild + republicação. Build local não basta.
  • O CI não builda a camada eBPF (opt-in, Linux-only); o controle que importa lá é "nenhum .bpf.o commitado".

Auto-checagem

O que o gate de self-audit verifica?

Actions fixadas por SHA, Cargo.lock commitado, ausência de .bpf.o, CVEs (cargo audit) e pentest aprovado (FAIL=0).

Por que embutir as denylists no binário?

Para torná-las tamper-proof: sem arquivo editável, um atacante não consegue afrouxar as regras de bloqueio.

Para que serve a atestação SLSA?

Para provar a proveniência do artefato (de qual fonte e build ele veio), dificultando substituição maliciosa.
