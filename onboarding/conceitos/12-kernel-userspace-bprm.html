Kernel vs userspace e o hook bprm_check_security

O kernel é a parte do SO que roda em modo privilegiado e media todo acesso a hardware e recursos. bprm_check_security é o ponto onde o kernel decide se um binário pode ser executado, e onde o Nemesis pode dizer não.

Em 30 segundos

Programas comuns rodam em userspace (sem privilégio direto). Para fazer qualquer coisa real (abrir arquivo, executar binário, abrir socket) eles chamam o kernel via syscall. Toda execução de binário passa por bprm_check_security no kernel; o Nemesis instala ali um programa eBPF que retorna -EPERM para comandos hostis.

Do zero

A CPU tem anéis de privilégio. O kernel roda no anel privilegiado; aplicações no anel restrito. A única forma de uma aplicação pedir algo privilegiado é a syscall, uma transição controlada para o kernel. Quando você roda perl payload.pl, a syscall execve dispara, e o kernel monta um linux_binprm (binary parameters) e consulta os módulos de segurança antes de transferir o controle ao novo programa. bprm_check_security é exatamente esse ponto de consulta.

Por que importa

Bloquear no kernel é qualitativamente mais forte que bloquear em userspace: não importa qual processo, shell, ou ofuscação o atacante use, a execução de um binário sempre passa por essa syscall. É o backstop que segura mesmo se o pretool estiver desligado (no Linux).

Como o Nemesis aplica

O programa eBPF anexa-se a lsm/bprm_check_security, lê o basename do binário, e se ele estiver no mapa de comandos bloqueados retorna -EPERM (operação não permitida):

📄 .nemesis/ebpf-kernel/ebpf/nemesis-block.bpf.c:69-138 
SEC("lsm/bprm_check_security")
int BPF_PROG(nemesis_check_exec, struct linux_binprm *bprm, int ret)
{
    if (ret != 0) return ret;              // respeita decisão anterior na cadeia
    // ... só processos no cgroup do agente são verificados ...
    filename = BPF_CORE_READ(bprm, filename);
    // extrai basename, busca no mapa blocked_commands
    blocked = bpf_map_lookup_elem(&blocked_commands, &key);
    if (!blocked) return 0;
    // publica evento no ring buffer, então:
    return -EPERM;
}

Note if (ret != 0) return ret;: o hook respeita decisões de LSMs anteriores na cadeia (stacking). E só processos no cgroup do agente são checados, para não afetar o resto do sistema.

Nuances e armadilhas

  • Esta camada só existe no Linux com BPF habilitado em /sys/kernel/security/lsm. No macOS não há contenção de kernel.
  • É opt-in: requer adicionar bpf à cmdline do GRUB. O comentário no topo do .bpf.c documenta isso.
  • O hook deliberadamente não usa bpf_override_return nem injeção de erro: usa o retorno legítimo do LSM (-EPERM), que é a forma suportada e estável.

Auto-checagem

O que é a estrutura linux_binprm?

"Binary parameters": a estrutura que o kernel monta em execve com os dados do binário a executar (nome, argumentos, ambiente). O hook lê o filename dela.

Por que if (ret != 0) return ret; no começo?

LSMs empilham (stacking). Se um módulo anterior já decidiu, o Nemesis respeita essa decisão em vez de sobrescrevê-la.

Bloquear aqui é mais forte que no pretool. Por quê?

Porque toda execução de binário passa por essa syscall no kernel, independente de processo ou ofuscação em userspace. O pretool pode ser contornado se desligado; o kernel não.
