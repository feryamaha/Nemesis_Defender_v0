Agentes LLM e o modelo de hooks pretool/posttool

Agentes de codificação executam ferramentas (ler, escrever, rodar comando) por conta própria. Os hooks são pontos onde código externo intercepta cada uso de ferramenta, antes (pretool) ou depois (posttool).

Em 30 segundos

Quando o agente decide "escrever este arquivo" ou "rodar este comando", a IDE chama o hook pretool com a ação pretendida. O hook pode permitir (exit 0) ou negar (exit 2). É o ponto de controle determinístico no laço do agente.

Do zero

Um agente LLM opera em laço: pensa, escolhe uma ferramenta, recebe o resultado, repete. As IDEs (Claude Code, Cursor, Devin, Codex…) expõem hooks: executáveis que recebem a ação pretendida e respondem com uma decisão. Pretool roda antes da ação (pode vetar); posttool roda depois (pode observar/reagir).

Por que importa

Sem o hook, a única defesa seria o próprio modelo "se comportar" — probabilístico. O hook insere um ponto de controle determinístico no laço, fora do modelo. É onde a tese do projeto se materializa.

Como o Nemesis aplica

O hook traduz o JSON de várias IDEs para um formato interno comum, e mapeia ferramentas para classes de ação:

📄 .nemesis/hooks/nemesis-pretool-check-unix.rs:312-347 — tradução multi-IDE 
let agent_action = match tool_name {
    "Edit" | "Write" | "MultiEdit" => "pre_write_code",
    "Bash" => "pre_run_command",
    "Read" | "Grep" => "pre_read_code",
    "apply_patch" => "pre_write_code",           // Codex
    "StrReplace" => "pre_write_code",            // Cursor
    s if s.starts_with("mcp__") => "pre_mcp_tool_use",
    _ => "pre_write_code",
};

O hook também garante que o daemon (e, no Linux, o eBPF) estejam de pé, de forma fire-and-forget, antes de avaliar a ação.

📄 .nemesis/hooks/nemesis-pretool-check-unix.rs:881-899 — ensure-daemon

Nuances e armadilhas

  • Há binários de hook por plataforma (unix/windows) e um conjunto de scaffolds por IDE (.claude/ .devin/ .cursor/ .codex/ .gemini/) que apontam para o mesmo motor.
  • O posttool existe (nemesis-posttool-check-unix), mas o enforcement primário é pretool: barato e antes do dano.
  • Em manutenção, o pretool fica desconectado de propósito; aí quem segura é o eBPF (Linux). No macOS, nada segura: invariante de segurança do projeto.

Auto-checagem

Qual a diferença entre pretool e posttool?

Pretool roda antes da ação e pode vetá-la; posttool roda depois e observa/reage. O Nemesis enforça no pretool.

Como o hook recebe a ação pretendida?

Como JSON pela stdin, enviado pela IDE; o hook traduz para um formato interno comum.

Por que isso é melhor que confiar no modelo?

Porque o hook é determinístico e externo ao modelo; não depende de o LLM "decidir" obedecer. Ver próxima página.
