Sockets, egress allowlist e prevenção de C2

Um socket é o endpoint de uma conexão de rede. Controlar a saída (egress) por uma allowlist é como o Nemesis impede exfiltração de dados e comunicação com servidores de comando e controle.

Em 30 segundos

Malware precisa "telefonar para casa": enviar dados roubados ou receber comandos (C2). Se o Nemesis bloqueia toda conexão exceto destinos explicitamente permitidos (allowlist deny-by-default), o canal de exfiltração/C2 morre. A decisão acontece no hook socket_connect do kernel.

Do zero

Quando um programa quer falar com um host, ele cria um socket e chama connect() com o endereço (IP + porta). Egress é o tráfego de saída. C2 (command and control) é a infraestrutura que o atacante usa para controlar máquinas comprometidas. Uma allowlist inverte a lógica usual: em vez de bloquear o que é ruim (impossível enumerar), permite só o que é bom e nega o resto.

Por que importa

É a manifestação técnica do invariante "100% local, sem rede externa". Se nada sai sem autorização, não há para onde exfiltrar credenciais nem de onde baixar payload. A allowlist vazia é o estado mais seguro: deny-by-default.

Como o Nemesis aplica

A semântica fail-closed está declarada na lógica pura de egress (testável sem kernel):

📄 .nemesis/ebpf-kernel/src/egress.rs:1-6 
//! egress.rs — lógica pura da allowlist de egress (CIDR:porta).
//! Semântica fail-closed: allowlist vazia ⇒ nenhuma conexão permitida (deny-by-default).

No kernel, o hook casa IP+porta contra o LPM trie; se não está na allowlist, emite evento e nega:

📄 .nemesis/ebpf-kernel/ebpf/nemesis-block.bpf.c:199-204 
struct egress_val *val = bpf_map_lookup_elem(&egress_allow_v4, &key);
if (val && (val->port == 0 || val->port == dport))
    return 0;       // destino allowlistado
egress_emit("ipv4");
return -EPERM;      // negado

A allowlist de destinos é configurada em denylist-ebpf/egress.toml. Em userspace, o motor de conteúdo também trata IMDS (169.254.169.254) e canais de rede como sinks de exfiltração. Ver exfil chain.

Nuances e armadilhas

  • port == 0 na regra significa "qualquer porta" para aquele CIDR; caso contrário casa porta exata.
  • O bloqueio de egress só vale no Linux com eBPF e modo enforce ligado. Em macOS, a prevenção de exfiltração se dá em userspace (detecção de chain no conteúdo).
  • Há uma suíte de pentest dedicada ao vetor de egress (pentest-nemesis-control/ebpf-kernel/VETOR-EGRESS.md).

Auto-checagem

O que é deny-by-default e por que allowlist vazia é segura?

Negar tudo exceto o explicitamente permitido. Allowlist vazia = nada sai = nenhum canal de exfiltração/C2.

O que significa port == 0 numa regra de egress?

"Qualquer porta" para aquele bloco CIDR; senão, a porta deve casar exatamente.

Como o egress se relaciona com o invariante "100% local"?

É o mecanismo que o impõe tecnicamente: sem saída autorizada, nenhuma feature pode depender de serviço externo.
