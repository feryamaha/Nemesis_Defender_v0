conceito 20 · Rust

Workspace, crates e por que Cargo.lock é commitado

Um workspace Cargo agrupa vários crates relacionados num só repositório com dependências resolvidas em conjunto. Commitar o lockfile é uma decisão de supply-chain, não de conveniência.

Em 30 segundos

O Nemesis é um workspace com quatro crates. O Cargo.lock grava a versão exata de cada dependência transitiva; commitá-lo garante que todos (e o CI) buildem os mesmos bytes. O self-audit do CI falha se o lockfile não estiver commitado.

Do zero

Um crate é a unidade de compilação do Rust (uma lib ou um binário). Um workspace compartilha um Cargo.lock e um diretório target/ entre múltiplos crates. O Cargo.toml declara dependências por faixa (ex.: regex = "1"); o Cargo.lock registra a versão exata escolhida (ex.: regex 1.10.4) com hashes.

Por que importa

Sem lockfile commitado, dois builds podem puxar versões diferentes de uma dependência, e um atacante que comprometa uma versão nova de um pacote transitivo entra silenciosamente. Com lockfile, a versão é fixa e qualquer mudança aparece no diff, revisável. Para uma ferramenta de segurança, build reprodutível é parte da postura.

Como o Nemesis aplica

O workspace declara seus quatro membros:

📄 .nemesis/Cargo.toml:1-8 
[workspace]
resolver = "2"
members = ["ast-linters", "ebpf-kernel", "nemesis-defender", "nemesis-doctor"]

[package]
name = "nemesis"
version = "8.2.0"

O CI exige o lockfile commitado e builda travado nele (--locked):

📄 .github/workflows/self-audit.yml:49-56, 73-76 
- name: Cargo.lock deve estar commitado (build reproduzivel / lock da supply chain)
  run: if [ -z "$(git ls-files .nemesis/Cargo.lock)" ]; then exit 1; fi
- name: Build do core (mesmos pacotes do release, travado na Cargo.lock)
  run: cargo build --release --locked -p nemesis -p nemesis-defender -p nemesis-doctor

Nuances e armadilhas

  • A regra clássica era "lib não commita lock, binário sim". Aqui o repositório produz binários distribuídos, então commitar é correto e exigido.
  • resolver = "2" muda como features são unificadas entre crates; importante em workspaces com targets distintos (ex.: o crate eBPF).
  • O --locked faz o build falhar se o lockfile precisaria mudar, garantindo que ninguém atualizou dependência sem commitar.

Auto-checagem

Por que commitar Cargo.lock num projeto de segurança?

Para build reprodutível e para que qualquer troca de versão transitiva apareça no diff e seja revisável — defesa de supply-chain.

O que --locked garante no CI?

Que o build usa exatamente o lockfile commitado; se ele precisaria mudar, o build falha.

Quantos crates compõem o workspace?

Quatro: ast-linters, ebpf-kernel, nemesis-defender, nemesis-doctor.
