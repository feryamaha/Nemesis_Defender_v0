Início/Rust/Panic e fail-closed

conceito 22 · Rust

Panic para fail-closed (por que falhar fechado é seguro)

Quando algo dá errado de forma inesperada, um sistema de segurança deve falhar negando, não permitindo. O Nemesis transforma qualquer pânico do hook em exit 2.

Em 30 segundos

Fail-open: se o controle quebra, a ação passa (inseguro). Fail-closed: se o controle quebra, a ação é negada (seguro). O hook do Nemesis embrulha tudo em catch_unwind; um pânico vira exit(2), ou seja, "bloqueado". Um bug nunca abre a porta.

Do zero

Em Rust, um panic é a forma de abortar diante de um estado impossível (índice fora do vetor, unwrap() em None). Por padrão ele desenrola a pilha e encerra a thread. std::panic::catch_unwind captura esse desenrolar para você decidir o que fazer. A escolha de design "qualquer pânico = negar" é o que chamamos de fail-closed.

Por que importa

Considere o cenário: um atacante envia um payload deliberadamente malformado que faz o hook panicar. Num sistema fail-open, o hook morre e a IDE executa a ação (atacante venceu). No Nemesis, o pânico é capturado e convertido em bloqueio: o atacante consegue, no máximo, um falso positivo, nunca um bypass.

Como o Nemesis aplica

📄 .nemesis/hooks/nemesis-pretool-check-unix.rs:861-879 — main fail-closed 
fn main() {
    match std::panic::catch_unwind(|| { run_pretool() }) {
        Ok(()) => {} // saiu normalmente via process::exit
        Err(panic) => {
            let msg = /* extrai mensagem do panic */;
            eprintln!("[NEMESIS ERROR] Hook panic: {}", msg);
            std::process::exit(2);   // fail-closed: pânico = bloqueado
        }
    }
}

O mesmo princípio aparece no fallback de shell, onde um pânico do hook vira exit 2 (documentado em hooks/nemesis-pretool-fallback.sh), e na semântica de egress (allowlist vazia = nada passa). Fail-closed é um tema, não um caso isolado.

Nuances e armadilhas

  • Fail-closed pode gerar falso positivo (bloquear algo legítimo por um bug). Isso é aceitável: o custo de um FP é reversível; o de um bypass não.
  • catch_unwind não captura abort (pânico configurado para abortar). O projeto usa o desenrolar padrão, compatível com a captura.
  • O contraste com o daemon: lá a ação irreversível (deletar) exige confiança alta; por isso o daemon não "fail-closes deletando", ele quarentena. Fail-closed se aplica ao bloqueio (reversível), não à destruição.

Auto-checagem

Fail-open vs fail-closed em uma frase?

Fail-open: quebrou, libera. Fail-closed: quebrou, nega. Segurança escolhe fail-closed.

O que acontece se um payload faz o hook panicar?

O catch_unwind captura e o processo sai com 2 (bloqueado). Sem bypass.

Por que o daemon não aplica fail-closed deletando?

Porque deletar é irreversível; ele quarentena (reversível) e exige revisão humana. Fail-closed vale para bloqueio, não destruição.
