nemesis-doctor (G1–G7) como diagnóstico e observabilidade

Uma defesa que você não consegue verificar é uma defesa em que você não pode confiar. O nemesis-doctor roda checagens de saúde da postura de proteção e reporta um veredito.

Em 30 segundos

nemesis-doctor é um binário de diagnóstico com sete grupos de checagem (G1–G7): compila, testa, inventaria binários, valida scaffolds de IDE, verifica o daemon, a camada eBPF e roda o pentest. --quick pula os grupos pesados.

Do zero

Observabilidade é a capacidade de entender o estado interno de um sistema pelo que ele expõe. Para uma ferramenta de segurança, isso significa responder: estou instalado corretamente? o daemon está no ar? o eBPF está ativo? as defesas ainda funcionam? O doctor automatiza essas perguntas em checagens reprodutíveis.

Por que importa

O invariante do projeto "saiba a postura de proteção antes de agir" depende de poder verificá-la. O doctor é a ferramenta que transforma "acho que está protegido" em "verifiquei: G1–G7 passaram".

Como o Nemesis aplica

O main orquestra todos os checks e sai com um código que reflete o resultado; --quick pula os grupos pesados:

📄 .nemesis/nemesis-doctor/src/main.rs:4-23 
let quick = args.iter().any(|a| a == "--quick");  // pula G1 compile, G2 testes, G7 pentest
let results = checks::run_all(quick);
let code = report::render(&results);
std::process::exit(code);

Os sete módulos de checagem são:

📄 .nemesis/nemesis-doctor/src/checks/mod.rs:3-9 
pub mod compile;    // G1 — compila o core
pub mod unit_tests; // G2 — testes unitários
pub mod inventory;  // G3 — binários presentes (bin/ ou target/release)
pub mod scaffold;   // G4 — scaffolds de hook por IDE
pub mod daemon;     // G5 — daemon ativo / ação
pub mod ebpf;       // G6 — camada eBPF (Linux)
pub mod pentest;    // G7 — suíte red-team

A resolução de caminhos sobe ao ancestral .nemesis (robusta a layout dev vs distro), mesma estratégia do daemon e do ledger.

📄 .nemesis/nemesis-doctor/src/checks/mod.rs:14-66 — nemesis_dir, binaries_dir

Nuances e armadilhas

  • Os rótulos G1, G2 e G7 (compile/testes/pentest) são os explicitamente nomeados no --quick; os demais grupos mapeiam para inventory/scaffold/daemon/ebpf conforme os módulos.
  • O doctor reflete a postura real: ele detecta o layout (distro .nemesis/bin/ tem precedência sobre target/release), evitando laudo errado.
  • Diagnóstico é leitura, não enforcement: o doctor não bloqueia ações, ele reporta. Complementa o enforcement, não o substitui.

Auto-checagem

O que --quick pula?

Os grupos pesados: G1 (compile), G2 (testes) e G7 (pentest).

Quantos grupos de checagem o doctor tem?

Sete módulos: compile, unit_tests, inventory, scaffold, daemon, ebpf, pentest.

Por que a resolução de caminho sobe ao ancestral .nemesis?

Para funcionar igual no layout de dev e no distribuído, sem assumir profundidade fixa de diretórios.
