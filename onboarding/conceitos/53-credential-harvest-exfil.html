Credential harvesting e exfiltration chains

Roubar segredos tem dois momentos: ler a credencial (harvest) e enviá-la para fora (exfil). O Nemesis trata a coexistência dos dois como malícia confirmada.

Em 30 segundos

Ler ~/.ssh/id_rsa ou process.env.AWS_SECRET_ACCESS_KEY é uma fonte. Mandar com curl/fetch para um host é um sink. Cada um isolado é, no máximo, suspeito; juntos no mesmo arquivo = exfil chain, malicioso. É a regra de primeira classe do exfil_chain.

Do zero

Credenciais vivem em lugares previsíveis: ~/.npmrc, ~/.aws/credentials, ~/.ssh/, variáveis de ambiente com TOKEN/SECRET/KEY, IMDS de nuvem (169.254.169.254). Harvesting é coletá-las; exfiltration é a chain que as envia para fora, por HTTP, DNS, WebSocket, netcat. A combinação é a assinatura comportamental do roubo.

Por que importa

Detectar só "leu um segredo" gera falso positivo (código legítimo lê segredos para usar em SDKs). Detectar só "fez um fetch" também. A coexistência fonte+sink é o que separa uso legítimo de exfiltração, e dá um sinal de alta confiança sem depender de heurística frágil.

Como o Nemesis aplica

O exfil_chain só dispara quando encontra uma fonte sensível e um sink de rede; fonte ou sink isolado não dispara:

📄 .nemesis/nemesis-defender/src/visitors/exfil_chain.rs:193-247 — scan_content 
let (source_evidence, source_label) = match source_match {
    Some(pair) => pair, None => return Vec::new(), // sem fonte → não é chain
};
let (sink_evidence, sink_label) = match sink_match {
    Some(pair) => pair, None => return Vec::new(), // sem sink → não é chain
};
// ambos presentes → exfil chain confirmado (Malicious)

Cuidado com env vars públicas: NEXT_PUBLIC_*, CLIENT_ID são expostas ao browser por design e não contam como fonte — isso evitou perda de dados em código Next.js legítimo:

Nuances e armadilhas

  • Serializar o ambiente inteiro (JSON.stringify(process.env) ou ...process.env) é tratado como fonte de exfil: não há uso legítimo de mandar todo o ambiente num payload.
  • Menção em prosa, sem ação, não dispara: o exfil_chain é determinístico, exige os dois padrões no código.
  • No Linux, o egress eBPF é a segunda linha: mesmo que o conteúdo escape, a conexão de saída pode ser negada. Ver egress.

Auto-checagem

Por que fonte isolada não basta para "malicious"?

Porque código legítimo lê segredos para usar em SDKs; sem um sink de saída, não há roubo.

Por que NEXT_PUBLIC_* é tratada como não-fonte?

Porque é pública por design (exposta ao browser); lê-la e enviá-la não é exfiltração. Ignorar isso causava perda de dados em código real.

O que torna o exfil_chain confirmatório?

A corroboração embutida: exige fonte de credencial + sink de rede coexistindo, padrão de alta confiança.
