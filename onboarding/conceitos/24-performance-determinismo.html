Performance e determinismo

O scanner roda no caminho crítico de cada ação do agente. Precisa ser rápido (não atrapalhar o fluxo) e determinístico (mesmo input, mesmo veredito), com limites que impedem abuso.

Em 30 segundos

O pipeline ordena estágios do mais barato (inspeção de bytes) ao mais caro (parse AST, decodificação recursiva). Limites duros impedem DoS: profundidade de decode máxima 3, tamanho máximo de candidato, faixas de comprimento. Determinismo significa que o veredito não depende de sorte, ordem de threads ou rede.

Do zero

Determinismo: dada a mesma entrada, a saída é sempre a mesma. Isso é o oposto de um modelo de IA (probabilístico). Para enforcement, é essencial: você precisa poder confiar e auditar a decisão. Performance num scanner não é só velocidade média; é também ter tetos garantidos para que entrada hostil não consuma recursos sem limite.

Por que importa

Se o scan fosse lento ou variável, ou o agente o contornaria (timeout) ou os usuários o desligariam (atrito). Se não tivesse tetos, um arquivo com encoding aninhado infinito poderia ser uma bomba de recursão. Determinismo é o que torna o pentest reprodutível e o gate de CI confiável.

Como o Nemesis aplica

A ordem dos estágios vai do barato ao caro, documentada e implementada:

📄 .nemesis/nemesis-defender/src/scanner/mod.rs:3-9 
//! Execution order (fastest to slowest):
//! 1. byte_scanner  2. entropy  3. regex_layer
//! 4. manifest_scanner  5. ast_scanner  6. decoder (recursive, max 3)

O decoder limita a recursão para impedir DoS por encoding aninhado:

📄 .nemesis/nemesis-defender/src/scanner/decoder.rs:22-28 
const MAX_DECODE_DEPTH: u8 = 3;       // impede DoS via encodings aninhados
const MIN_DECODED_LEN: usize = 8;     // não revarre lixo curto
const MAX_CANDIDATE_LEN: usize = 4096; // não tenta decodificar blobs enormes

A entropia também limita a faixa de strings analisadas (20–500 chars), evitando custo em texto irrelevante.

📄 .nemesis/nemesis-defender/src/scanner/entropy.rs:14-16

Nuances e armadilhas

  • O AST só roda para linguagens suportadas; arquivos desconhecidos ficam em bytes+regex, mais baratos. É uma otimização e uma decisão de corretude (não há parser para o que não se conhece).
  • Determinismo exige cuidado com estruturas não ordenadas: o pretool ordena os arquivos JSON de denylist por nome antes de iterar, para que o "primeiro hit" seja estável.
  • Os tetos podem causar falso negativo teórico (payload no 4º nível de encode). É um trade-off consciente: profundidade ilimitada seria um vetor de DoS pior.

Auto-checagem

Por que ordenar os estágios do barato ao caro?

Para que a maioria dos arquivos seja decidida pelos estágios rápidos, reservando o parse/decoder caros para quando necessário.

O que impede uma bomba de recursão no decoder?

MAX_DECODE_DEPTH = 3 mais limites de tamanho mínimo/máximo de candidato.

Por que determinismo importa para o CI?

Porque o pentest precisa ser reprodutível: mesmo input, mesmo resultado, para o gate falhar/passar de forma confiável.
