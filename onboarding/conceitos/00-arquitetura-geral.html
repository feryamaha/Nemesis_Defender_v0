Arquitetura geral do Nemesis Defender

Como as peças se encaixam: três camadas de defesa, um pipeline de scan de oito estágios e um modelo de decisão que só deleta quando a hostilidade é confirmada.

Em 30 segundos

O Nemesis intercepta cada ação de um agente de IA antes dela acontecer (hook pretool), escaneia o conteúdo/comando, e nega com exit 2 se for hostil. Um daemon observa o sistema de arquivos e move malware confirmado para a quarentena. No Linux, uma camada eBPF opcional segura exec e rede no próprio kernel. Tudo 100% local.

Por que importa

As camadas 1 e 2 usam o mesmo motor de conteúdo, então o veredito de segurança é idêntico para bytes idênticos, seja na hora da escrita (pretool) ou no sistema de arquivos (daemon). Isso é declarado e verificado no próprio código.

Como o Nemesis aplica

O orquestrador é a função scan_content. Ela executa os estágios em ordem e delega a decisão final a compute_severity.

pub fn scan_content(path: &Path, content: &[u8]) -> DefenderResult {
    if is_path_excluded(path) { /* early return Clean */ }
    let language = detect_language(path);
    let mut all_violations = Vec::new();
    // Layer 1: byte (BiDi/PUA/homoglyph/zero-width)
    // Layer 2: entropia (Shannon)
    // Layer 3: regex (denylist-defender.json)
    // Layer 4: manifesto · 4.5 ide_config · 4.6 exfil_chain
    // Layer 5: AST (tree-sitter)  · Layer 6: decoder recursivo (máx 3)
    all_violations.retain(|v| !allowlist_loader::is_allowlisted(&v.evidence));
    let severity = compute_severity(&all_violations);
    DefenderResult { severity, violations: all_violations, scan_depth, path, language }
}

A arquitetura de crates é um workspace Rust com quatro membros: nemesis-defender (motor + daemon + CLI), ebpf-kernel (Linux), nemesis-doctor (diagnóstico) e ast-linters (pausado).

📄 .nemesis/Cargo.toml — workspace, pacote nemesis v8.2.0

Nuances e armadilhas

  • A ordem comentada em scanner/mod.rs difere da ordem realmente executada em lib.rs. O código é a fonte da verdade: a execução real tem 8 chamadas de estágio (byte tem 4 sub-scans).
  • O daemon é security-only: regras de qualidade nunca chegam nele, porque qualidade ⇒ bloquear escrita (reversível), nunca deletar (irreversível). Ver severidade/quarentena.
  • No macOS não há camada de kernel: com o pretool desligado, nada contém. No Linux, o eBPF cobre.

Auto-checagem

As camadas pretool e daemon podem divergir no veredito de segurança?

Não para o mesmo conteúdo: ambas chamam scan_content, cuja fonte de regras de segurança é única (denylist-defender.json via regex_layer). A única divergência legítima é a camada de qualidade, exclusiva do pretool.

Quantos crates há no workspace e qual a versão do pacote raiz?

Quatro crates (ast-linters, ebpf-kernel, nemesis-defender, nemesis-doctor); o pacote raiz nemesis está na versão 8.2.0 (Cargo.toml).

O que acontece se o caminho está na lista de exclusão?

scan_content retorna Clean imediatamente (early return), sem escanear. É como o estado interno de .git/ e as pastas de pentest evitam falsos positivos.
