Capabilities, cgroups v2 e Landlock (sandbox sem root)

Três mecanismos do Linux para limitar o que um processo pode fazer: capabilities fatiam o poder de root, cgroups agrupam e identificam processos, e Landlock sandboxa sem precisar de root.

Em 30 segundos

O Nemesis usa cgroup id para saber quais processos são do agente (e só checá-los no kernel). Usa Landlock para se auto-restringir sem root, via no_new_privs + restrict_self. Capabilities são o modelo de privilégio que torna isso possível sem ser root.

Do zero

Capabilities: em vez do binário root/não-root, o Linux divide os poderes de root em peças (ex.: CAP_NET_ADMIN, CAP_SYS_ADMIN). Um processo pode ter só o que precisa. cgroups v2: agrupam processos para controle de recursos e contabilidade; cada cgroup tem um id único. Landlock: um LSM que deixa um processo não privilegiado impor a si mesmo restrições de acesso a arquivos e exec, que se propagam aos filhos e não podem ser revertidas.

Por que importa

Sandbox sem root é crucial para uma ferramenta que roda na máquina do dev: pedir root seria um risco maior que o que se quer mitigar. Landlock dá contenção forte com privilégio zero. O cgroup id é o que permite ao eBPF mirar só o agente, sem afetar o sistema todo.

Como o Nemesis aplica

Landlock exige PR_SET_NO_NEW_PRIVS antes de restrict_self — é o que dispensa a capability de admin:

pub fn apply_sandbox(allowed_exec_paths: &[&str]) -> LandlockStatus {
    // no_new_privs: obrigatório p/ landlock_restrict_self sem CAP_SYS_ADMIN
    let r = unsafe { libc::prctl(libc::PR_SET_NO_NEW_PRIVS, 1, 0, 0, 0) };
    if r != 0 { return LandlockStatus::Error("prctl PR_SET_NO_NEW_PRIVS failed".into()); }

Nuances e armadilhas

  • O código degrada por ABI: em kernels mais antigos, usa a versão de Landlock disponível, em vez de falhar.
  • Se o cgroup do agente não está configurado (id 0), o hook eBPF permite tudo — fail-open nessa identificação é deliberado para não quebrar o sistema; o enforcement de conteúdo (pretool/daemon) é a defesa primária.
  • Há também um módulo seccomp.rs: filtra syscalls permitidas, complementar ao Landlock (que filtra acesso a arquivos/exec).

Auto-checagem

Por que no_new_privs é obrigatório antes de Landlock?

Porque permite que um processo sem privilégio aplique landlock_restrict_self sem CAP_SYS_ADMIN; sandbox sem root.

Para que serve o cgroup id no eBPF?

Para identificar quais processos são do agente e checar só eles, deixando o resto do sistema intocado.

Capability vs root: qual a diferença?

Root é tudo-ou-nada; capabilities fatiam esse poder, então um processo recebe só o privilégio específico de que precisa.
