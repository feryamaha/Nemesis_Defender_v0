Entropia de Shannon na detecção de payload ofuscado

Entropia mede a imprevisibilidade de uma sequência de bytes. Texto codificado ou cifrado tem entropia alta; código normal não. Esse contraste vira um sinal de detecção.

Em 30 segundos

Uma string base64 ou cifrada parece "aleatória": cada caractere é quase equiprovável, então a entropia é alta. Código e texto normais têm padrões (letras frequentes, palavras), logo entropia menor. O Nemesis sinaliza strings com entropia acima de 5,5 bits/char como possível payload codificado.

Do zero

A entropia de Shannon de uma sequência é H = -Σ p(x)·log₂ p(x), somando sobre os símbolos presentes. Ela vale 0 quando tudo é o mesmo byte e até 8 bits/char quando todos os 256 valores são equiprováveis. Intuição: quanto mais imprevisível o próximo byte, maior a entropia. Dados comprimidos, cifrados ou codificados se aproximam do máximo.

Por que importa

Atacantes escondem comandos em strings codificadas para escapar de assinaturas. Mas a própria codificação deixa um rastro estatístico: alta entropia. É um sinal independente da assinatura, e independência é o que dá poder ao modelo de corroboração (2 sinais distintos = malicioso).

Como o Nemesis aplica

O cálculo é a fórmula de Shannon, direta sobre os bytes:

📄 .nemesis/nemesis-defender/src/scanner/entropy.rs:19-40 — shannon_entropy 
fn shannon_entropy(data: &[u8]) -> f64 {
    let mut freq = [0u32; 256];
    for &b in data { freq[b as usize] += 1; }
    let len = data.len() as f64;
    let mut entropy = 0.0_f64;
    for &count in freq.iter() {
        if count > 0 {
            let p = count as f64 / len;
            entropy -= p * p.log2();
        }
    }
    entropy
}

O limiar é calibrado para evitar falsos positivos de coisas legítimas (classes Tailwind têm entropia ~4,9–5,1):

📄 .nemesis/nemesis-defender/src/scanner/entropy.rs:14-16 
const ENTROPY_THRESHOLD: f64 = 5.5;   // acima disso = suspeito de payload codificado
const MIN_STRING_LEN: usize = 20;
const MAX_STRING_LEN: usize = 500;

O visitor emite o sinal high_entropy, que é corroborante (Tier B): sozinho vira só Suspicious; combinado com outro sinal distinto, Malicious. Ver corroboração.

Nuances e armadilhas

  • Entropia alta não prova malícia: um hash, um UUID ou uma chave pública também são alta entropia. Por isso é corroborante, não confirmatório.
  • O limiar 5,5 é um trade-off calibrado contra falsos positivos reais (Tailwind). Calibrar limiares contra dados legítimos é parte da engenharia.
  • Só strings de 20–500 chars entram: curtas demais não têm sinal estatístico; longas demais custam e raramente são "comandos".

Auto-checagem

Por que base64 tem entropia alta?

Porque distribui os caracteres de forma quase uniforme no alfabeto de 64 símbolos, tornando o próximo char imprevisível.

Por que high_entropy é corroborante e não confirmatório?

Porque hashes/UUIDs/chaves legítimas também têm alta entropia; sozinho geraria falsos positivos, então exige um segundo sinal.

Por que o limiar é 5,5 e não, digamos, 5,0?

Para ficar acima da entropia de coisas legítimas como classes Tailwind (~4,9–5,1) e ainda pegar payloads codificados (>5,5).
