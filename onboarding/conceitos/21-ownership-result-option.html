Ownership, borrowing e Result/Option

Rust garante segurança de memória sem coletor de lixo, e modela ausência e erro como valores explícitos no tipo. Isso elimina classes inteiras de bugs que uma ferramenta de segurança não pode se dar ao luxo de ter.

Em 30 segundos

Em Rust, cada valor tem um dono; quando o dono sai de escopo, a memória é liberada — sem GC, sem use-after-free. Ausência é Option<T> (Some/None); erro é Result<T,E> (Ok/Err). O compilador obriga a tratar os dois casos. O Nemesis usa isso para tratar input hostil/malformado sem crashar.

Do zero

Ownership: um valor tem um único dono; passá-lo move a posse. Borrowing: você pode emprestar uma referência (&T imutável ou &mut T mutável), e o borrow checker garante em tempo de compilação que não há acessos conflitantes nem ponteiros pendentes. Option/Result: em vez de null ou exceções, a possibilidade de "nada" ou "erro" vive no tipo, e ignorá-la não compila.

Por que importa

Um scanner processa entrada não confiável o tempo todo (conteúdo de arquivos, JSON de IDEs, payloads decodificados). Em C, um parse malformado vira corrupção de memória. Em Rust, vira um Err ou None que o código é obrigado a tratar. Segurança de memória é pré-requisito de uma ferramenta de segurança.

Como o Nemesis aplica

Decodificação que pode falhar retorna Option; o chamador trata a ausência sem panicar:

📄 .nemesis/nemesis-defender/src/scanner/decoder.rs:315-339 — try_decode_base64 
fn try_decode_base64(s: &str) -> Option {
    let cleaned: String = s.chars().filter(|c| !c.is_whitespace()).collect();
    if cleaned.len() < 8 { return None; }
    match BASE64.decode(&cleaned) {
        Ok(bytes) => match String::from_utf8(bytes) {
            Ok(s) if !s.trim().is_empty() => Some(s),
            _ => None,
        },
        Err(_) => None,
    }
}

Leitura de arquivo retorna Result, e o erro é convertido em saída controlada, nunca em crash:

📄 .nemesis/nemesis-defender/src/main.rs:149-155 — fs::read → Err tratado 
let content = match std::fs::read(&path) {
    Ok(c) => c,
    Err(e) => { eprintln!("Cannot read {}: {}", path.display(), e); std::process::exit(1); }
};

Nuances e armadilhas

  • O Nemesis evita unwrap() em caminhos de dados não confiáveis, preferindo match/?. Onde usa expect(), é em invariantes do próprio binário (ex.: obter o próprio caminho), e ainda assim coberto pelo catch_unwind.
  • String::from_utf8_lossy é usado para nunca rejeitar bytes inválidos no input da IDE; já em conteúdo de scan, from_utf8 falha cedo e o estágio simplesmente pula (binário não é código).
  • O borrow checker às vezes exige clonar (ex.: tainted_vars.clone() no taint tracker) — custo consciente em troca de segurança.

Auto-checagem

Como Rust evita use-after-free sem GC?

Pelo sistema de ownership/borrowing: o compilador prova que nenhuma referência sobrevive ao dono.

Por que Option/Result em vez de null/exceções?

Porque tornam ausência e erro explícitos no tipo; o compilador obriga a tratar, eliminando o esquecimento.

Por que o decoder retorna Option<String>?

Porque uma string pode não ser base64 válido; None sinaliza "não decodificável" sem panicar.
