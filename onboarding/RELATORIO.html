Relatório técnico do material

Resumo do inventário forense que originou este material, as decisões de design adotadas, e as limitações declaradas com honestidade.

1. Inventário (lido do código)

Workspace

Workspace Rust (resolver = "2"), pacote raiz nemesis v8.2.0. Quatro crates: nemesis-defender (motor + daemon + CLI), ebpf-kernel (Linux, opt-in), nemesis-doctor (diagnóstico), ast-linters (pausado). Oito binários de hook declarados em [[bin]] (pretool unix/windows, posttool, lsp, etc.).

Pipeline de scan (ordem real em lib.rs::scan_content)

  1. Byte: scan_bidi, scan_pua, scan_homoglyphs, scan_zero_width
  2. Entropia (Shannon, limiar 5,5)
  3. Regex (fonte única denylist-defender.json)
  4. Manifesto · 4.5 IDE poisoning · 4.6 Exfil chain
  5. AST tree-sitter (JS/TS/Bash/Python)
  6. Decoder recursivo (máx 3 níveis)

Seguido de allowlist (supressão por evidência) e compute_severity.

Modelo de severidade

Três níveis (Clean/Suspicious/Malicious). Tier A confirmatório (1 hit = Malicious) e Tier B corroborante (1 tipo = Suspicious; 2+ tipos distintos = Malicious). Confirmado → quarentena (reversível), não deleção.

Camada eBPF

Dois hooks BPF-LSM: bprm_check_security (exec) e socket_connect (egress allowlist LPM). Escopo por cgroup id; modo enforce/observação; Landlock (sem root), seccomp. Linux-only, opt-in.

DevSecOps

self-audit.yml: pin de Actions por SHA, Cargo.lock commitado, sem .bpf.o, cargo audit, gate de pentest (FAIL=0). release.yml: build + atestação SLSA. CODEOWNERS. Denylists de bloqueio embutidas no binário via include_str!.

2. Decisões de design do material

  • Forensic-first: cada afirmação sobre o Nemesis cita arquivo (e, quando útil, faixa de linhas). Onde o enunciado divergiu do código, o código venceu (ver cobertura).
  • Trechos curtos: 5–25 linhas por conceito, fiéis ao repo, comentados. Decisão confirmada na calibração inicial.
  • Agrupamentos: decode→exec + reverse shell numa página; self-clean + time-gated + unicode steg noutra (mesma família Tier B / byte). Um conceito obrigatório = uma página.
  • Offline e sem CDN: CSS único, JS próprio (busca local + TOC + realce mínimo de sintaxe), diagramas em SVG inline. Abre com duplo clique no index.html.
  • Acessibilidade: headings semânticos, breadcrumb, skip-link, contraste AA, navegação por teclado (/ foca a busca), aria-label em diagramas.
  • Sem travessões longos no corpo do texto, conforme pedido.

4. Limitações e TODOs do material

Honestidade
  • Trechos de código são excertos fiéis mas elididos (com /* ... */) para caber no formato; o arquivo-fonte é sempre citado para conferência.
  • Números voláteis (contagem de casos de pentest, linhas exatas do SLSA no release) foram deliberadamente não fixados para não defasar; aponta-se o arquivo.
  • Alguns conceitos do SO (capabilities, cgroups v2) são parcialmente evidenciados: o código os usa indiretamente (cgroup_id, no_new_privs); a página explica o conceito e marca a evidência indireta.
  • O realce de sintaxe em JS é um tokenizador mínimo próprio (keywords/strings/comentários/números), não um parser completo. Suficiente para legibilidade, não para correção sintática.
  • A camada eBPF foi lida da fonte (.bpf.c + Rust); não foi compilada nem executada como parte deste material.

5. Como verificar

Qualquer citação pode ser conferida abrindo o arquivo indicado no repositório (caminho relativo à raiz). Os caminhos e faixas de linha refletem o estado do código no momento da análise; pequenas variações de linha podem ocorrer após edições futuras, mas as funções/constantes citadas são âncoras estáveis.
